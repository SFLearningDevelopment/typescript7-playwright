Module 1

TypeScript 7 for Playwright Testers

native compiler tsc --noEmit type-check gate

If you write Playwright tests in TypeScript, here's what TypeScript 7 means for you — and one fact about Playwright that makes it matter more than you might expect. Open each idea.

native compiler Same TypeScript, ~10× faster +

TypeScript 7 is not a new version of the language — it's the compiler, rebuilt to run as native code (ported to Go). It's often about 10× faster than the previous version, while checking your types in exactly the same way. Nothing about how you write a Playwright test changes; the tool that checks it just got much faster.

For a small handful of spec files you might not notice. For a large suite — hundreds of specs, page objects, fixtures, and helpers — the difference between a type-check that takes a minute and one that takes a few seconds is felt every day, both in CI and in your editor.

the key fact Playwright does not type-check your tests +

This is the fact that makes TypeScript 7 directly relevant to you. When you run npx playwright test, Playwright takes your .spec.ts files, transforms them to JavaScript, and runs them — it does not check your types. A spec with a real type error (a misspelled property, a wrong argument) will still run; the type mistake simply isn't caught at that step.

So where does type safety come from? From running the TypeScript compiler separately, as its own step: tsc --noEmit. That command checks your types without producing any output (Playwright does the actual running). This separate gate is the thing TypeScript 7 makes ~10× faster — which is exactly why a faster compiler is a real win for a Playwright tester, not an abstract one.

why it helps Faster gate, faster editor, fewer flaky surprises +

Three concrete benefits. First, your pretest type-check (the tsc --noEmit gate) finishes far faster, so the check you run before every test run — locally and in CI — stops being a bottleneck. Second, your editor uses the same engine, so autocomplete, hover, and error squiggles get snappier in big test repos. Third, catching type errors quickly means fewer silly mistakes (a wrong locator method, a typo'd fixture) slip through to a slow browser run only to fail there.

The rest of this course shows how to install it, wire the type-check gate, type your page objects and fixtures well, and adopt it on your suite without disrupting your existing runs.

Refresh your knowledge?

Why use TypeScript for Playwright at all? Playwright runs fine with plain JavaScript, so what does TypeScript add? Type safety — the compiler catches whole categories of mistakes before a browser ever opens: a misspelled method (page.getbyrole instead of getByRole), passing the wrong type of argument, using a fixture that doesn't exist. In a test suite this matters doubly, because a typo that survives to runtime wastes a slow end-to-end run to discover something the compiler could have told you instantly.

What the compiler actually does. The TypeScript compiler reads your .ts files, checks that the types line up (this is "type-checking"), and can also produce plain JavaScript ("emit"). For Playwright tests you usually only want the checking, not the output — Playwright handles turning your specs into runnable JavaScript itself. That's why the command you'll run is tsc --noEmit: check the types, emit nothing.

Spec, page object, fixture — the vocabulary. A spec is a test file (login.spec.ts). A page object is a class that wraps a page's locators and actions so tests read cleanly and don't repeat selectors. A fixture is Playwright's mechanism for providing set-up things to a test (a logged-in page, test data) in a reusable way. TypeScript types all three, so when you use them wrong, you find out at your desk — and TypeScript 7 makes finding out faster.

Accurate as of June 2026 The package and binary names below track the TypeScript 7.0 RC. Preview details can shift — if a command behaves differently, check the official TypeScript site and the @typescript/native-preview npm page.

What you need first: Node.js

Playwright already runs on Node.js, so you likely have it. The native TypeScript 7 compiler needs Node.js 16.20.0 or newer — any current LTS (20 or 22) is fine. Check it:

node -v   # expect v16.20.0 or higher
npm -v

Install the TypeScript 7 compiler

During the preview, the native compiler ships under its own package name, so it won't clash with anything Playwright installed. Add it to your test project as a dev dependency:

npm install -D @typescript/native-preview
Binary name. On the 7.0 RC and later the command is tsc (same as always). On earlier preview builds it was tsgo. If tsc isn't found right after installing a preview, try tsgo — same compiler.

Run your first type-check

From your Playwright project, type-check without producing output. Playwright picks up your tsconfig.json; tsc does too:

# check types only — Playwright still runs the tests separately
npx tsc --noEmit

# point at a specific tsconfig
npx tsc -p tsconfig.json --noEmit

# watch mode for local development
npx tsc -p tsconfig.json --noEmit -w

If your specs are clean, you'll get no output — that's success. Here's a clean spec the gate passes silently:

// tests/login.spec.ts — clean: types line up
import { test, expect } from '@playwright/test';

test('user sees the dashboard after sign in', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('User Name').fill('alice');
  await page.getByLabel('Password').fill('secret');
  await page.getByRole('button', { name: 'Sign in' }).click();
  await expect(page.getByText('Welcome, alice')).toBeVisible();
});
$ npx tsc --noEmit
$                       # no output, exit code 0 = types are clean

Now introduce a deliberate mistake — a misspelled method (getByRol) and a wrong argument type (a number where a string is expected):

// tests/login.spec.ts — broken on purpose
import { test, expect } from '@playwright/test';

test('user sees the dashboard after sign in', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('User Name').fill(12345);          // number, not string
  await page.getByLabel('Password').fill('secret');
  await page.getByRol('button', { name: 'Sign in' }).click();  // typo
  await expect(page.getByText('Welcome, alice')).toBeVisible();
});

The gate catches both before any browser opens:

$ npx tsc --noEmit
tests/login.spec.ts:5:39 - error TS2345: Argument of type 'number'
  is not assignable to parameter of type 'string'.

5   await page.getByLabel('User Name').fill(12345);
                                            ~~~~~

tests/login.spec.ts:7:14 - error TS2339: Property 'getByRol' does not
  exist on type 'Page'. Did you mean 'getByRole'?

7   await page.getByRol('button', { name: 'Sign in' }).click();
               ~~~~~~~~

Found 2 errors in the same file, starting at tests/login.spec.ts:5
Without the gate, both mistakes would survive to playwright test — the typo'd method would throw a confusing runtime error minutes into a browser run, and the number argument might coerce silently. TypeScript 7 makes running this check fast enough to do on every save.

The pattern you're building toward

The whole point is to run this fast check before your tests. A typical setup wires it as a "pretest" step so it runs automatically — we build this out in Module 2:

// package.json (preview)
{
  "scripts": {
    "pretest": "tsc --noEmit",
    "test": "playwright test"
  }
}

With this, npm test type-checks first and only runs the browser tests if the types are clean — and on TypeScript 7 that first step is fast enough that it never feels like a tax.

Refresh your knowledge?

npm, npx, and dev dependencies. npm installs packages your project lists in package.json. npx runs a command from a package, preferring the copy installed in your project — so npx tsc runs the compiler version your project pinned, not a random global one. Installing with -D (--save-dev) records it under devDependencies: things needed while developing and testing, but not shipped to production. The compiler belongs there, exactly like Playwright itself.

What --noEmit means. Normally a compiler produces output files. --noEmit tells tsc to do all its type-checking but write nothing to disk. For Playwright tests that's precisely what you want: you're using the compiler as a checker, not a build tool, because Playwright already turns your specs into runnable JavaScript on its own. Check, report problems, emit nothing.

The pre script convention. npm has a small built-in convention: if you have a script named test, a script named pretest runs automatically right before it. That's how a single npm test can first run your type-check and then your Playwright run, in order, with no extra command. It's the standard way teams make "check types, then test" a one-step habit.

What tsc is. tsc is the TypeScript compiler — the command-line program that reads your .ts files, checks the types, and (unless you pass --noEmit) writes out JavaScript. When this course says "the gate," it means running tsc --noEmit over your tests. It's a separate tool from Playwright: Playwright runs your tests, tsc checks them. Both happen to read the same tsconfig.json.

What tsconfig.json is. It's the configuration file that tells tsc how to treat your project — which files to include, how strict to be, what (if anything) to emit, and how to resolve imports. The compiler picks it up automatically from your project. You'll meet it properly in Module 2, where it turns out Playwright reads only a few of its options while tsc reads all of them. For now, just know it's the dial-box for the type-check gate.

tsc vs. tsgo — what a tester should know. You may see the native compiler invoked as tsgo in some preview docs. Here's the whole story: tsgo was the temporary command name used by earlier native-preview builds (the betas and nightlies), to keep it clearly separate from the classic tsc while both could be installed. From the TypeScript 7.0 RC onward, the native compiler uses the familiar tsc name — so your scripts and habits carry over unchanged. They are the same compiler; only the command name differed by build phase. Practical rule: use tsc; if you're on an older preview build and tsc isn't found, try tsgo. Nothing about your tests changes either way.
