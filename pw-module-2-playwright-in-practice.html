Module 2

Playwright on TypeScript 7 in Practice

page objects fixtures tsconfig

Module 1 covered why a faster compiler matters and how to run it. This module is the practical core: how to wire the type-check gate properly, what Playwright actually does with your tsconfig, and how to type your tests so the compiler catches mistakes for you. Open each idea.

the gate Type-check before you test +

Because Playwright runs specs without type-checking them, the discipline that makes TypeScript worthwhile is running tsc --noEmit as a gate before the browser tests. Here's why that matters — this spec has a real type error, but npx playwright test will happily run it:

// login.spec.ts — the wrong argument type is a bug Playwright won't catch
import { test } from '@playwright/test';

test('sign in', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('Password').fill(12345);  // number, not string
});

Playwright transforms and runs it; the mistake only surfaces (if at all) mid-run in the browser. The separate gate catches it first, before any browser opens:

$ npx tsc --noEmit
login.spec.ts:5:41 - error TS2345: Argument of type 'number'
  is not assignable to parameter of type 'string'.

Wire it as a pretest script so a single npm test does both in order — check types, then run. (npm runs a pretest script automatically right before test.) Run npm test on your own machine and the type-check happens first:

// package.json
{
  "scripts": {
    "pretest": "tsc --noEmit",
    "test": "playwright test"
  }
}
$ npm test
> pretest: tsc --noEmit      # runs first; stops here if types fail
> test: playwright test      # only runs if the gate passed

In CI, run the same check as its own step before playwright test, so a type error fails the pipeline fast and cheap — no browser spin-up needed to find it:

# GitHub Actions
- name: Type-check
  run: npx tsc --noEmit
- name: Run Playwright tests
  run: npx playwright test

On TypeScript 7 this gate is fast enough that it stops being something people skip "to save time." That's the real behavioural win: a fast gate is a gate people actually keep.

tsconfig Playwright honours only a few tsconfig options +

Important and easy to miss: when Playwright loads your test files, it only respects a small subset of tsconfig.json options — allowJs, baseUrl, paths, and references. Everything else in your tsconfig (like strict, target, the full type-checking settings) is used by tsc when it checks your types, not by Playwright when it runs them.

So there are really two consumers of your tsconfig: tsc --noEmit (which reads all of it, and is where TypeScript 7's speed applies) and Playwright's runner (which reads just those four). Keeping that split clear avoids confusion like "why didn't Playwright respect my strict setting?" — it isn't supposed to; your tsc gate is.

typing well Well-typed page objects and fixtures pay off +

The value you get from the fast gate is proportional to how well your tests are typed. If your page objects, fixtures, and test data carry good types, the compiler catches a wrong method name, a missing property, or a mismatched argument instantly — and on TypeScript 7 it does so fast enough to run on every save.

The "Typing your tests" tab walks through the patterns: typed page objects, Playwright's typed custom fixtures, and typed test data. None of this is new in TypeScript 7 — but a faster compiler is what makes leaning on these types comfortable in a large suite.

Refresh your knowledge?

The page object pattern. A page object is a class that represents a page (or a part of one) in your app, bundling its locators and the actions you take on it. Instead of repeating page.getByLabel('Password') across twenty specs, you write a LoginPage class with a login(user, pass) method and call that. Tests read like plain English, and when the UI changes you fix the selector in one place. TypeScript types the class so callers can't misuse it.

What a Playwright fixture is. A fixture is how Playwright supplies ready-made things to a test — the built-in page is a fixture, and you can define your own (a logged-in page, a seeded user, an API client). Fixtures are set up before the test and torn down after, automatically. Playwright lets you extend the base test with typed custom fixtures, so each test receives exactly the typed helpers it asked for.

Why types matter more in tests than you'd think. A type error in application code might be caught by a unit test. A type error in a test often isn't caught by anything until the test runs in a real browser and fails confusingly minutes later — or worse, passes for the wrong reason. The tsc gate is your unit test for your tests, and a fast compiler is what lets you run it constantly rather than occasionally.

A typed page object

A page object is just a typed class. Notice that the constructor takes a typed Page, and methods have typed parameters — so a caller who passes the wrong thing is caught by tsc:

import { type Page, type Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly username: Locator;
  readonly password: Locator;
  readonly submit: Locator;

  constructor(page: Page) {
    this.page = page;
    this.username = page.getByLabel('User Name');
    this.password = page.getByLabel('Password');
    this.submit   = page.getByRole('button', { name: 'Sign in' });
  }

  async login(user: string, pass: string): Promise<void> {
    await this.username.fill(user);
    await this.password.fill(pass);
    await this.submit.click();
  }
}
What the gate catches here. Call login(123) instead of login('alice','pw') and tsc --noEmit flags it immediately — wrong argument type, too few arguments. Misspell getByRole as getByrole and it's caught too. None of this would fail at playwright test time until the browser actually ran.

A typed custom fixture

Playwright lets you extend the base test with your own fixtures, fully typed. Here a loginPage fixture hands every test a ready LoginPage:

import { test as base, expect } from '@playwright/test';
import { LoginPage } from './pages/LoginPage';

type MyFixtures = { loginPage: LoginPage };

export const test = base.extend<MyFixtures>({
  loginPage: async ({ page }, use) => {
    await page.goto('/login');
    await use(new LoginPage(page));
  },
});

export { expect };

Now a spec asks for the fixture by name and receives it typed:

import { test, expect } from './fixtures';

test('user can sign in', async ({ loginPage, page }) => {
  await loginPage.login('alice', 'secret');
  await expect(page.getByText('Welcome, alice')).toBeVisible();
});
Type-safe by construction. Because MyFixtures declares loginPage: LoginPage, a test that typos the fixture name or uses a method the class doesn't have is caught by the gate — not discovered during a browser run.

Typed test data

Give your test data a type so a malformed record is caught where it's defined, not deep in a failing assertion:

type User = { name: string; email: string; admin: boolean };

const users: User[] = [
  { name: 'Alice', email: 'alice@example.com', admin: true  },
  { name: 'Bob',   email: 'bob@example.com',   admin: false },
  // { name: 'Cara', email: 'cara@example.com' }  // tsc: missing 'admin'
];

for (const u of users) {
  test(`sign in as ${u.name}`, async ({ loginPage }) => {
    await loginPage.login(u.email, 'secret');
  });
}
Refresh your knowledge?

Types, interfaces, and Locator/Page. Playwright ships its own types: Page is the type of a browser page, Locator is the type of a located element, and so on. When you write readonly username: Locator, you're telling the compiler what that property holds, so it can check every use of it. A type alias (type User = {...}) names a shape so you can reuse it; an interface does much the same for object shapes. Either lets the compiler hold your data to a contract.

What Promise<void> and async mean here. Playwright actions are asynchronous — they return a Promise, a value that completes later. async/await is how you write that readably: await this.submit.click() waits for the click to finish. A method typed Promise<void> promises to complete but return no value. Getting these wrong (forgetting an await) is a classic test bug; with proper typing and a linter rule, the gate can catch many of them.

Generics, briefly — the <MyFixtures> part. base.extend<MyFixtures>(...) uses a generic: you hand extend a type describing your fixtures, and it gives you back a test function that knows about them. Generics are how a reusable function preserves specific type information instead of falling back to "anything." You don't need to write generics often as a tester, but recognising the angle-bracket syntax helps you read Playwright's typed APIs.

Accurate as of June 2026 The tsconfig behaviour below reflects Playwright's documented support and the TypeScript 7.0 RC. Verify against the official Playwright TypeScript docs and the @typescript/native-preview npm page if something differs.

Two readers, one tsconfig

Your tsconfig.json is read by two different things, and they use different parts of it:

ReaderUsesRole
tsc --noEmitAll compiler options (strict, target, etc.)Type-checks your tests. This is where TypeScript 7 speeds things up.
Playwright runnerOnly allowJs, baseUrl, paths, referencesTransforms and runs specs (no type-checking).

So your strict setting, your target, your full type rules — those govern the gate, run by tsc. Playwright ignores them when running. Knowing this prevents the common "Playwright didn't enforce my tsconfig" confusion.

A sensible tests tsconfig

Playwright recommends a separate tsconfig.json inside your tests directory so test settings don't bleed into app code. A reasonable starting point:

// tests/tsconfig.json
{
  "compilerOptions": {
    "strict": true,            // catch null/any mistakes in tests
    "target": "ESNext",
    "module": "commonjs",
    "paths": {                 // Playwright honours this; no baseUrl needed in TS7
      "@pages/*": ["./pages/*"]
    }
  }
}

Playwright automatically picks up the closest tsconfig.json for each file it loads, so a tests/tsconfig.json applies to your specs without extra wiring.

Wire the gate: local and CI

Locally, the pretest script makes npm test check then run:

{
  "scripts": {
    "pretest": "tsc -p tests/tsconfig.json --noEmit",
    "test": "playwright test"
  }
}

In CI, run the type-check as its own step before the tests, so a type error fails fast and cheap (no browser spin-up needed to find it):

# GitHub Actions (sketch)
- name: Type-check
  run: npx tsc -p tests/tsconfig.json --noEmit
- name: Run Playwright tests
  run: npx playwright test
TypeScript 7 angle. Because the native compiler is ~10× faster, that "Type-check" CI step adds very little wall-clock time — so there's no excuse to drop it for speed. A fast gate is one you keep in the pipeline.

A note on TS7 defaults

TypeScript 7 starts from modern defaults (we cover these properly in Module 3). For a test project the ones to know are strict on by default and the types array no longer auto-loading every @types package. If a global suddenly reads as "cannot find name," list what you need — most commonly:

{
  "compilerOptions": {
    "types": ["node"]   // plus any others your tests rely on
  }
}
Refresh your knowledge?

What tsconfig.json is. It's the file that tells the TypeScript compiler how to treat your project: which files to include, how strict to be, what to emit. The compiler reads it automatically. For a Playwright project it does double duty — tsc reads all of it for the gate, and Playwright reads a few options from it when loading specs.

baseUrl and paths. These let you write tidy imports. paths maps a prefix to a location — so import { LoginPage } from '@pages/LoginPage' can resolve to tests/pages/LoginPage instead of a long relative path. paths is among the few options Playwright actually honours, which is why it's worth knowing as a tester. Older TypeScript required baseUrl to be set alongside paths, but TypeScript 7 deprecates baseUrl — modern TypeScript resolves paths relative to the tsconfig.json's own location, so you use paths on its own. (If you set baseUrl under TS7 you'll get a deprecation error.)

Separate tsconfig per directory. TypeScript lets each directory have its own tsconfig.json, and a child config can extends a parent to inherit and override settings. A common layout is a root config for shared settings and a tests/tsconfig.json for test-specific preferences. Playwright looks up the closest config to each file, so this layering "just works" for your specs.

Read the pipeline

Open .github/workflows/ci.yml. The important part is the order of steps:

- name: Install dependencies
  run: npm install

- name: Type-check          # the gate — runs FIRST
  run: npx tsgo -p tests/tsconfig.json --noEmit

- name: Install Playwright browsers
  run: npx playwright install --with-deps chromium

- name: Run Playwright tests # only if the gate passed
  run: npx playwright test

This is the whole philosophy of the course in one file: because Playwright doesn't type-check your specs, the pipeline runs the fast type-check gate before spending time on browsers. A type error fails the build in seconds, not minutes.

The three problems — walkthrough

Work through these in order. Each is a real failure mode, its cause, and the fix. Try to solve each yourself first.

Part 1 — the install step fails

What you'll see. The very first CI step fails before anything else runs:

npm error code EUSAGE
The `npm ci` command can only install with an existing package-lock.json

Why. The workflow uses npm ci, the "clean install" command — but it requires a committed package-lock.json, and this project doesn't have one. npm ci is stricter than npm install by design.

Fix. In .github/workflows/ci.yml, change the install step from npm ci to npm install (which resolves from package.json and needs no lockfile). Commit, and CI moves past install.

- name: Install dependencies
    run: npm ci
    run: npm install

Lesson: infrastructure matters. Before any test logic runs, the pipeline has to be able to install your project.

Part 2 — the type-check step runs the wrong compiler

What you'll see. Install now passes, but the type-check step fails oddly — it downloads and runs a random, ancient package:

npm warn exec The following package was not found and will be installed: tsc@2.0.4
This is not the tsc command you are looking for

Why. This is the exact tsc-vs-tsgo point from Module 1. The project uses the native TypeScript 7 compiler (@typescript/native-preview), whose binary on preview builds is tsgo, not tsc. So npx tsc doesn't find a local tsc, goes to the registry, and grabs an unrelated old package named tsc.

Fix. Use tsgo in both the CI step and the package.json scripts, so the gate runs the native compiler you actually installed.

# .github/workflows/ci.yml
- name: Type-check
    run: npx tsc  -p tests/tsconfig.json --noEmit
    run: npx tsgo -p tests/tsconfig.json --noEmit
// package.json — same change in the scripts
"pretest":   "tsc  -p tests/tsconfig.json --noEmit" "tsgo -p tests/tsconfig.json --noEmit",
"typecheck": "tsc  -p tests/tsconfig.json --noEmit" "tsgo -p tests/tsconfig.json --noEmit"

Lesson: knowing the compiler's real binary name isn't trivia — get it wrong and your gate silently runs the wrong thing.

Part 3 — types pass, but the browser tests fail

What you'll see. The type-check now passes cleanly — and yet the browser tests fail. One test passes (the one that just checks the empty list); the two that add a todo time out:

2 failed
  [chromium] › todo.spec.ts › can add a single todo
  [chromium] › todo.spec.ts › can add several todos
1 passed
   at TodoPage.add (tests/pages/TodoPage.ts:24)  # timed out on .fill()

Why. This is the most important lesson in the whole course. The type-check gate passed — the types were correct all along — but there was still a real runtime bug. The TodoMVC demo is a hash-routed single-page app; the page object navigated with goto('/') and started typing before the app was ready, so fill() timed out. Types can't catch that. Only running the test can.

Fix. Navigate to the app's actual route and wait for the input before interacting, and align baseURL so the path resolves correctly.

// tests/pages/TodoPage.ts
async goto(): Promise<void> {
  await this.page.goto('/');
  await this.page.goto('/todomvc/#/');
  await this.newTodoInput.waitFor({ state: 'visible' });
}
// playwright.config.ts — baseURL points at the site root so '/todomvc/#/' resolves
use: {
  baseURL: 'https://demo.playwright.dev/todomvc',
  baseURL: 'https://demo.playwright.dev',
}

Lesson — the one to remember past the exam: a passing type-check is necessary but not sufficient. The gate protects you from a whole class of mistakes, but it cannot tell you your test actually works against a real app. That's why you still run the browser tests — and why the pipeline runs the gate first (fast, cheap) and the tests after (slower, but the real proof).
