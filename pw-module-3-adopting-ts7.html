Module 3

Adopting It on Your Suite

breaking changes CI coexistence

You know what TypeScript 7 is and how to use it on your tests. This last module is about adopting it on a real suite without disruption: the few breaking changes a test project actually meets, and how to bring it into CI safely alongside your existing setup. Open each idea.

low risk For a test suite, the risk is low +

Here's the reassuring part. A test project doesn't ship production JavaScript — you use TypeScript only to check your specs, and Playwright does the running. So the riskiest part of a TypeScript 7 migration for application teams (changes to how production code is emitted) simply doesn't apply to you. What's left is a short list of breaking changes that affect type-checking, and they're easy to handle.

Because the type-checking behaviour itself is unchanged from the previous version, a test suite that type-checks cleanly today should type-check cleanly on TypeScript 7 — once you've reconciled a couple of changed defaults. That's the whole migration for most Playwright projects.

two changes Two defaults are the ones you'll meet +

TypeScript 7 starts from modern defaults. For a Playwright project, two are worth knowing because they're the ones likely to produce an error on first run: strict is on by default, and the types array no longer auto-loads every @types package. Both have simple, explicit fixes, covered in the next tab.

Most other breaking changes (dropped legacy module formats, old resolution modes) rarely touch a modern Playwright setup, which already uses current settings. So you're reconciling a couple of things, not rewriting a config.

safe rollout Introduce it beside what you have +

The safe way to adopt is the same parallel-change discipline good teams use for any tooling switch: add TypeScript 7 alongside your current compiler, prove they agree, then switch. In practice that means an informational 7.0 type-check job in CI next to your existing one, the native preview extension for whoever wants faster editing, and side-by-side install so nothing breaks if you need to fall back.

Nothing here is risky or irreversible — which is exactly why a test suite is a great place to adopt TypeScript 7 early. The Rollout tab gives you the concrete steps.

Refresh your knowledge?

What "breaking change" and "default" mean. A breaking change is a change that can make something which used to work stop working — here, a project that compiled before might now report an error. A default is the value a setting takes when you don't specify it. Most of TypeScript 7's "breaking changes" for testers are really changed defaults: the compiler now assumes a different value than before when your tsconfig stays silent. The fix is almost always to state the value you want explicitly — and explicit settings are immune to default changes.

What strict is. strict is an umbrella setting that turns on a family of stricter type checks at once — most notably catching values that might be null/undefined and flagging variables whose type the compiler couldn't infer (implicit any). Good test code usually already satisfies it; if turning it on surfaces errors, each one is a real "this might be undefined" worth fixing. You can also opt back out with "strict": false while you work through them.

What @types packages are. Many JavaScript libraries ship their type information separately, in packages under the @types/ scope (for example @types/node gives you types for Node.js built-ins). Older TypeScript auto-loaded all of them into global scope; TypeScript 7 doesn't, so you list the ones your tests actually use via the types option. That's why a global that "just worked" before might need @types/node named explicitly now.

strict on by default

If your tests project never set strict, TypeScript 7 now enables it. You'll see errors like "Object is possibly 'undefined'" where a locator or fixture result might not exist. These are real — handle the missing case:

// strictNullChecks flags this:
const row = page.getByRole('row').first();
const text = (await row.textContent()).trim();   // textContent() may be null
const text = (await row.textContent() ?? '').trim(); // handle null

If your suite has many such spots and you want to land the migration first, opt out explicitly and tighten later:

{
  "compilerOptions": {
    "strict": false   // temporary; re-enable and fix incrementally
  }
}

types no longer auto-loads @types

Previously every @types/* package in node_modules was loaded into global scope automatically. TypeScript 7 loads none by default, so a global like process (from Node) can suddenly read as undefined. Name what you depend on:

{
  "compilerOptions": {
    "types": ["node"]   // add others your tests use, e.g. a custom global
  }
}
Playwright's own types come from @playwright/test, which you import directly (import { test } from '@playwright/test') rather than relying on global auto-loading — so Playwright's types are unaffected by this change. It's mainly ambient globals like Node's that need listing.

The "is my project ready?" check

The simplest readiness test: get your suite type-checking cleanly under your current compiler first, with strict explicitly set the way you want and no deprecated options. If it's clean there, TypeScript 7 should agree. Run the gate and confirm:

npx tsc -p tests/tsconfig.json --noEmit

A clean result here means the move to TypeScript 7 is essentially just installing the faster compiler.

Refresh your knowledge?

strictNullChecks, in plain terms. Part of the strict family, this check stops null and undefined from being silently treated as valid for every type. In tests this shows up a lot, because many Playwright calls can legitimately return "nothing found" — textContent() can be null, a .first() match might not exist. The check forces you to handle that case (with ??, an if, or an assertion), which is exactly the kind of bug that otherwise produces a confusing runtime failure mid-test.

The nullish-coalescing operator ??. a ?? b evaluates to a unless a is null or undefined, in which case it gives b. It's the tidy way to supply a fallback: (await row.textContent() ?? '') means "the text, or an empty string if there wasn't any." It's the most common one-character-pair fix you'll apply when turning on strict checks.

Ambient globals vs. imported types. Some types are imported where you use them (import { Page } from '@playwright/test') — these are unaffected by the types default because you name them at the point of use. Others are ambient globals: names available everywhere without import, like process or console's richer Node typings, supplied by an @types package. The types array controls which ambient global packages load — so it's globals, not your imports, that the new default affects.

Step 1 — Install side by side

Add the native preview alongside your existing TypeScript so nothing breaks and you can compare:

# your current compiler stays…
npm install -D typescript
# …and the native preview joins it
npm install -D @typescript/native-preview

They install together without conflict. You now have both compilers available in the project.

Step 2 — Add an informational 7.0 gate in CI

Don't replace your existing check — add a 7.0 type-check job next to it, marked non-blocking at first, so it reports on real specs without failing the pipeline:

# GitHub Actions (sketch)
- name: Type-check (current — authoritative)
  run: npx tsc -p tests/tsconfig.json --noEmit

- name: Type-check (TS7 — informational)
  run: npx tsc -p tests/tsconfig.json --noEmit
  continue-on-error: true      # non-blocking while you build trust

- name: Run Playwright tests
  run: npx playwright test

Watch the informational job across a few weeks of real commits. When it consistently agrees with your current check, it has earned trust.

Step 3 — Let editors opt in

Have a few testers install the TypeScript Native Preview extension for VS Code and use it day-to-day. The editor is where the speed is most felt on a big suite, and it's zero-risk: it changes how fast the editor responds, not what gets committed or run. Collect any feedback on behaviour differences.

Step 4 — Make 7.0 the gate

Once the informational job has agreed for a while, promote it: make TypeScript 7 the authoritative type-check, drop the continue-on-error, and point your local pretest at it. Keep your previous compiler installed as a fallback. Your suite now gets the fast gate everywhere:

{
  "scripts": {
    "pretest": "tsc -p tests/tsconfig.json --noEmit",   // now TS7
    "test": "playwright test"
  }
}
Why this order works. Every step is reversible and low-stakes: the informational job can't break your pipeline, the editor change affects nobody downstream, and the previous compiler stays available. You prove agreement before you depend on it — the same parallel-change discipline that makes any tooling switch calm.

Refresh your knowledge?

What CI is, for a tester. Continuous integration (CI) is the automated system — GitHub Actions, GitLab CI, Jenkins — that runs your checks and tests on a clean server every time code is pushed. For test automation it's home turf: it's where your Playwright suite already runs on every change. Adding a type-check step there means every push is gated on clean types before the (slower) browser tests run, catching type mistakes in seconds instead of minutes.

What "non-blocking" / continue-on-error means. A CI step can be set so that even if it fails, the overall pipeline still passes — it reports a result without gating anything. That's the safe way to trial a new tool: run the TypeScript 7 check as non-blocking, watch whether it ever disagrees with your trusted check, and only make it blocking once you're confident. It's a trial run with no consequences.

Why "side by side, then switch" is the safe pattern. Installing the new compiler next to the old one — rather than replacing it — means you always have a known-good fallback and can compare the two on identical code. You switch only after they've agreed across real commits. This parallel-change approach is why adopting a brand-new compiler on your suite can be low-drama: at no point are you betting the pipeline on something unproven.
